## Project Overview

Bachelor's thesis (deadline 2026-06-20): *Autonomous navigation using Reinforcement Learning*. A DQN agent drives a Pioneer 3-AT robot autonomously through an **unstructured environment** in the **Webots R2025a** simulator. Navigation is **pure RL on sensors** — the learned policy decides everything from LiDAR + IMU + relative goal position; there is no hand-crafted geometric path planner.

The world is a **150×150 m irregular-terrain map**: an **organic, irregularly-shaped navigable clearing** (clearing outline varies ~45–54 m with angle via `boundary_r(angle)`, **not a circle**) of gentle rolling hills (real pitch/roll, ~10° max, height ≈ ±1.25 m, with a flat spawn zone in the center) set in a **pine forest whose density increases toward the map edge** — there is **no ring of trees around the clearing and no raised rim/mountains/rocks**. Trees start sparse just outside the clearing and **thicken outward** into a **fine, dense, impassable belt at the square perimeter**, so the robot **cannot fall off the map**. The robot is bounded physically by these tree collisions (flood-fill-verified: from the centre it cannot reach the map edge) and by an **out-of-bounds episode termination at the map edge** (`max(|x|,|y|) > HALF − bounds_margin`), plus rollover termination on steep terrain. **Goals are always sampled inside the central clearing (`boundary_r`)**, so the forest is a graded obstacle field/soft-to-hard boundary the policy learns to avoid. The terrain is an `ElevationGrid` whose heights are **baked from a Python terrain model** (the `terrain_height` family of functions lives in `scout_env.py` and is imported by `generate_world.py`), so the RL height queries equal the rendered/colliding surface exactly. The interior is deliberately **unstructured (not a village)**: just **2 utility structures** (a warehouse + a barn) on **flat graded pads** (the terrain is leveled under each so they sit flush, never floating/sunk), plus several **inaccessible zones** — densely-packed boulder fields, a pine thicket, a container pile and a rubble heap that the robot must drive around — and scattered off-road clutter (rocks, fallen logs, barrels, crates, abandoned vehicles). The robot spawns at `terrain_height + 0.03` so its wheels rest on the surface (the Pioneer's wheel contact is at its own origin).

## How to Run

**Prerequisites:** Webots R2025a installed; Python venv activated.

```powershell
.\rl_env\Scripts\activate
# (Re)build the world from the terrain model + object scatter (offline, not in Webots):
cd diploma_webots_project\controllers\rl_scout
python generate_world.py     # writes ../../worlds/world.wbt
# Then open diploma_webots_project/worlds/world.wbt in Webots; it launches the rl_scout controller.
```

`world.wbt` is **generated** — do not hand-edit the terrain. To reshape the ground, edit the terrain model **in `scout_env.py`** (`terrain_height` / `HILL_AMP`; clearing shape via `boundary_r` / `BOUNDARY_R0` / `BOUNDARY_AMP`; building positions + pads via `BUILDINGS` / `PAD_*`) or `generate_world.py` (object scatter counts; forest gradient `FOREST_GRID`/`FOREST_TAU`; edge belt `EDGE_BELT_W`/`EDGE_BELT_GRID`), then re-run `python generate_world.py`. You **may** still add or move *objects* directly in the Webots editor — `scan_world_obstacles()` reads them dynamically. First load needs internet to fetch/cache the GitHub PROTOs.

> ⚠️ **The current `worlds/world.wbt` has been HAND-CUSTOMIZED in Webots** (the user added houses, a windmill, swing, vehicles, props, and reshaped obstacle layout; Claude later edited it directly — replaced cube crates with natural rocks/logs, added a Debris Area + narrow passages). **It is no longer purely generated.** Re-running `generate_world.py` **overwrites all of that** with the bare generated scene. So: **do not regenerate without first backing up `world.wbt`** (a `world.wbt.bak` snapshot exists). The terrain model (`scout_env.py`) still matches the world's terrain, and `scan_world_obstacles()` keeps the RL in sync with whatever objects are present, so the hand-customized world trains fine. To make permanent terrain/shape changes now, hand-edit `world.wbt` (or merge changes back into the generator deliberately).

Startup Tkinter menu (ttk-themed, with editable **Episoade / Pași-ep / Eval-N** fields):
- **Antrenează de la zero** — fresh DQN (configurable episodes/steps; goal-distance curriculum; CSV metrics). **Start + goal are both random each episode** (`reset(random_start=True)` samples a navigable, obstacle-clear, gentle-slope start anywhere in the clearing), so the policy generalises from any pose, not just centre. After training it **auto-runs two batch evals**: random-start (primary → `eval_summary.csv`) **and** centre-start (`eval_summary_center.csv`, the thesis comparison figure).
- **Continuă antrenarea** — resumes from `dqn_model.pth`/`_best.pth` (loads q+target weights, `epsilon=RESUME_EPSILON=0.3`, continues episode numbering by appending to `training_metrics.csv`).
- **Evaluare automată (N episoade)** — N episodes hands-free, **random start + goal**, `epsilon=0`; prints success / collision / rollover / out-of-bounds / **stuck** / timeout rates + avg steps-to-goal; (over)writes `eval_summary.csv` + first 20 trajectories → feeds `plot_metrics.py`.
- **Evaluare comparativă (start centru)** — same as above but **start fixed at centre** (θ + goal random) → writes `eval_summary_center.csv` + `eval_traj_center_*.csv` for the thesis center-vs-random comparison figure.
- **Evaluare manuală (start/goal)** — prompts for start/goal, logs trajectories; press `g` to change goal live.
- **Baseline reactiv (fără DQN)** — reactive LiDAR+goal controller, single manual episode.
- **Baseline reactiv — N episoade** — the same reactive controller run hands-free for N random-start/random-goal episodes → `eval_summary_baseline.csv` + `eval_traj_baseline_*.csv`; feeds the DQN-vs-reactive comparison figure (`eval_comparison.png`). No model needed.
- **Test rapid (50 episoade)** — short from-scratch run to sanity-check that reward rises.

Generate thesis figures after a run (offline, not inside Webots):
```powershell
cd diploma_webots_project\controllers\rl_scout
python plot_metrics.py   # reads logs/*.csv → writes figures/*.png
```

## Architecture

All logic in `diploma_webots_project/controllers/rl_scout/`:

### [scout_env.py](diploma_webots_project/controllers/rl_scout/scout_env.py) — RL environment
- **Dynamic environment reading:** `scan_world_obstacles()` walks the Webots scene tree via the Supervisor at every `reset()` and builds the obstacle list (x, y, radius) automatically. Objects are matched by PROTO type name (`PROTO_RADIUS`), by exact `DEF OBS_*` (`DEF_RADIUS`), or by `DEF OBS_*` **prefix** (`DEF_PREFIX_RADIUS`, for auto-numbered generated obstacles like `OBS_CONTAINER0`). This list is used **only for safe goal placement** — collision avoidance itself is learned from LiDAR. Add objects to the world freely without touching Python.
- **Terrain (single source of truth, defined here):** the `terrain_height(x,y)` / `terrain_slope(x,y)` / `in_navigable(x,y)` / `boundary_r(angle)` / `heightfield_grid()` functions and map constants (`MAP_SIZE=150`, `GRID_DIM=76`, `BOUNDARY_R0=50`, `BOUNDARY_AMP=5`, `ARENA_LIMIT = BOUNDARY_R0+BOUNDARY_AMP = 55` (the *max* navigable radius), `HILL_AMP`, `BUILDINGS`/`PAD_*`) live at the top of this file and are **imported by `generate_world.py`**, so the baked `ElevationGrid` equals the heights the RL queries. **`boundary_r(angle)`** is the irregular-**clearing** outline (smooth sum of low harmonics, radius ~45–54 m) — the **single source of truth for the goal/clearing region**: `in_navigable` and `_clamp_to_arena` key off it, so **goals are always sampled inside the clearing** (never in the forest). **Flat building pads:** `terrain_height` levels a smooth (smoothstep) flat platform under each entry of `BUILDINGS` so large flat-based buildings sit flush instead of floating/sinking on the hills. `_surface_z_at` places the robot start and goal disk on the real surface. Goals are sampled only on traversable ground (`in_navigable` uses `boundary_r(angle)` + `terrain_slope < max_goal_slope`). The **robot**, by contrast, is bounded by the surrounding forest (sparse near the clearing → impassable at the map edge); the `out_of_bounds` termination is the safety net at the **map edge** (`max(|x|,|y|) > HALF − bounds_margin`), so it cannot fall off the map. The goal marker is a **floating beacon** (emissive orb + thin beam) whose `DEF GOAL_TARGET` node sits on the surface (`goal_marker_z=0`) while its children hover at a **constant height above the terrain** — terrain-independent, so it never clips/buries on slopes (the old flat disk did) — and it stays **above the LiDAR cone**, so the agent still finds it from the goal-relative state, not the LiDAR. Terrain is **rolling hills everywhere** (sum of sinusoids × `HILL_AMP`, no rim/basin/mountains) with a flat central spawn zone + flat building pads.
- **State (ACTIVE = run 5 config: 23 dims = `FRAME_STACK=1` × 23, `_stack_frames` returns the single current frame, `_get_state` builds it):** `[0:12]` **12 uniform LiDAR sector minima covering the FULL 180° FOV** (15° each, min-pooling; `N_LIDAR_SECTORS=12`), `[12:15]` frontal vertical profile (upper/mid layer + up−down difference), `[15:23]` goal distance & angle, v/ω, z, roll, pitch, yaw-rate. **LiDAR range `LIDAR_MAX_RANGE=10` m** — must equal `maxRange` in BOTH `.wbt` worlds; all LiDAR features normalize to it. **Actions: 5** — 0 fwd, 1/2 fwd-turn, 3/4 rotate; −1 is the baseline's internal reverse. History: point rays (run 2) → 5 frontal sectors ±56° (runs 3–4) → full-FOV 12 sectors (run 5, HEADLINE) → **ablation: stacking ×3 + reverse + 20 m range (run 6) and the same minus reverse + bigger net (run 7) BOTH regressed → reverted to the run-5 lean config.** The `FRAME_STACK`/`LIDAR_MAX_RANGE` knobs remain (set to 1/10) so the ablation is reproducible. **Every state-layout change invalidates earlier models — retrain.** Reward reads the current frame (front = sectors 5–6, flanks = 3,4,7,8; collision = min of all 12) and its **proximity/collision thresholds are in METERS** (`front_m = front·LIDAR_MAX_RANGE`; 1.0/0.5 m penalties, 0.25 m collision) → at 10 m these equal run-5's original normalized thresholds (0.10/0.05/0.025) but are now range-invariant. **Webots lidar range-image order is LEFT→RIGHT** ("pixels are stored in scan lines running from left to right", Lidar reference manual) → **low sector indices = robot's LEFT**; the pre-2026-06-11 left/right feature labels were swapped (irrelevant to the DQN, but the old baseline turned toward the more-blocked side — part of its 69% stuck).
- **`world_to_robot_frame(wx, wy)`** — homogeneous world→robot transform used to compute the goal-relative distance/angle in the state.
- **LiDAR 2D convolution:** `_conv2d_smooth` applies a separable [1,2,1] smoothing kernel to the layered range image before feature extraction (sensor denoising).
- **Reward (`_compute_reward`), redesigned 2026-06-10 after run-3 reward hacking:** progress-to-goal shaping (×20) is the **ONLY positive per-step term** — potential-based (Ng et al.), anti-symmetric, so closed loops sum to 0 and it cannot be farmed. The old flat bonuses (orientation +1/step, free-space +0.2/step) were **REMOVED**: at γ=0.99 any flat income r is worth r/(1−γ)=100·r, so the +1 orientation bonus was worth ~100 vs ~35 discounted value of reaching a far goal → run 3 learned to PARK facing the goal (53% stuck, stops in open field). Progress-gated variants are still farmable (small circles give positive progress half the time AND evade the anti-stuck anchor, which resets at 0.5 m). Remaining terms (all ≤0): misalignment −0.05·|angle|; LiDAR proximity penalties; instability penalty; living cost; **idle cost −0.02/step while v<0.1**; |w| penalty. Terminals: +200 goal; −20 collision; −20 rollover (|roll|/|pitch| > `max_safe_roll/pitch`); −20 out-of-bounds; **−20 anti-stuck** (was −5) when the robot moves < `stuck_move_thresh`=0.5 m from its anchor for `stuck_patience`=600 steps. **Unit-tested offline in `tests/test_reward.py` (11 scenarios — run it after any reward edit). Reward changes invalidate run-3 models → retrain (run 4). Expect the new reward SCALE to be much lower (successful episode ≈ +20·distance + 200, i.e. hundreds not thousands).**
- **Random start (`_sample_free_start`):** when `reset(random_start=True)`, the start pose is sampled anywhere in the clearing — navigable (`in_navigable` with `start_clearance`=1.4), gentle slope (`< max_goal_slope`), and clear of all scanned obstacles. Obstacles are scanned at the **top** of `reset()` (they're static) so both start and goal sampling avoid them. Fallback start = centre (always flat/free).

### [dqn_agent.py](diploma_webots_project/controllers/rl_scout/dqn_agent.py) — DQN
Dueling QNetwork (**shared 128→128 → value & advantage streams →64**, the run-5 headline config; `QNetwork.HIDDEN/STREAM`. Run 7 tried 256/128 for the 69-dim stacked state → caused value-function divergence (loss 6→1246) → reverted), Double DQN, **replay buffer 200k stored as float32** (50k ≈ only ~36 episodes of history at ~1400 steps/ep — rare far-goal successes were overwritten before being exploited), batch 64, target update every 500 steps, Adam lr=5e-4 with a **floored LR schedule** (`LambdaLR`, 5e-4 → floor 1e-4 around ep.~450). The old `StepLR(5000, 0.9)` ticked every *train step* (~1300 steps/episode → ×0.9 every ~4 episodes), so **lr ≈ 0 by ep.~300 — the 2026-06-09 run's avg50 peaked at ep.226 and then froze for 700 episodes**; that was the main convergence killer. **Epsilon decays PER EPISODE** (`decay_epsilon()` called once/episode, `epsilon_decay=0.993` → ε≈0.24 at ep.200, ≈0.06 at ep.400, hits 0.01 ≈ep.650) — the previous per-step decay collapsed ε to the floor by ~ep.55 (random episodes hit max steps, burning the exploration budget before the curriculum introduced far goals). `train_step()` returns the TD-loss (logged per episode). **CPU is correct here** — the net is tiny, so GPU per-step overhead would be slower, and Webots (not the NN) is the wall-clock bottleneck.

### [rl_scout.py](diploma_webots_project/controllers/rl_scout/rl_scout.py) — entry point
Webots Supervisor setup, training/eval/baseline loops, Tkinter menus, and CSV logging to `logs/`. `MAX_STEPS=6500` (208 s sim — 5000/160 s was physically too short for >60 m goals with detours; stuck episodes are still cut early by the anti-stuck rule) and the goal-distance curriculum (`CURRICULUM_*`, scaled to `arena_limit`) are sized for the large map. `run_training`/`run_eval_batch` now **propagate max_steps into `env.max_steps`** (previously the env's internal 5000 silently capped whatever the menu said). **`MODEL_BEST` is saved by best avg50** (≥50 episodes), not by best single episode (a lucky long episode used to win); evals load `MODEL_BEST`. `run_eval_batch(policy_fn=...)` also drives the **reactive-baseline batch eval** (menu button) → `eval_summary_baseline.csv` (the baseline reads the 12-sector state; re-run 2026-06-12 with the corrected L/R halves: **31% success / 64% stuck / 4% collision**). **"Evaluare pe hartă nevăzută"** menu button = same random-start eval but tagged `_unseen` — open `worlds/world_eval.wbt` in Webots first (generalization test on an unseen obstacle layout). Each eval dumps its own `obstacles{tag}.csv` so trajectory figures draw the right map. `NUM_EPISODES=1200` (run 5 headline config; best-avg50 checkpointing handles the late-degradation pathology). The baseline's front threshold is in meters (`front_m < 1.2`) → invariant to the lidar-range change, so the measured 31% remains the valid thesis reference. The Supervisor reads optional devices (`imu/gps/gyro/accelerometer`) with graceful fallback, but the generated world always provides them.

### [generate_world.py](diploma_webots_project/controllers/rl_scout/generate_world.py) — world generator (offline)
CLI: `--seed N --out FILE.wbt [--force]`. **It REFUSES to overwrite the hand-customized `world.wbt` without `--force`.** `python generate_world.py --seed 77 --out world_eval.wbt` produced **`worlds/world_eval.wbt`** — a fresh obstacle layout on the IDENTICAL terrain (same terrain model import), used for the unseen-map generalization eval. **Imports the terrain model from `scout_env.py`** (`terrain_height`, `boundary_r`, `heightfield_grid`, `terrain_slope`, `BUILDINGS`, constants) so the baked grid + map shape match the RL. Emits: the EXTERNPROTO block (GitHub R2025a scheme for objects, local `../protos/Pioneer3at.proto` for the robot), the `DEF TERRAIN Solid` with the `ElevationGrid` (geometry + `boundingObject` via `USE TERRAIN_EG`, `Grass` appearance), deterministic (`SEED`) layout: the 2 buildings at the **deterministic `BUILDINGS` positions on their flat pads** (sit flush; no overlaps — audit-verified), a few abandoned vehicles/street-furniture near them (vehicles have physics so they settle), **5 inaccessible zones** (`boulder_field`/`thicket`/`container_cluster`/`rubble_pile` — densely-packed obstacle clusters the robot can't enter, audit-verified), and trees/rocks/**fallen logs/barrels/crates/containers** scattered over the interior (placement is **boundary-aware** via `boundary_r`, clear of the start zone and kept `FOREST_CLEAR` inside the forest). **Unique DEFs:** every generated `OBS_*` gets a global unique id (`_uid()`) → no duplicate DEF names (which Webots rejects). **Correct vertical placement:** real PROTOs with a centered origin get an explicit z-offset (`OilBarrel` +height/2 = 0.44; `Rock` +0.05·scale) so they rest on the ground; debris is emitted as **custom `Solid` nodes** (`crate_node` Box, `log_node` lying Cylinder, `container_node`/`rubble_pile` Box) with z computed from `terrain_height` + half-extent → full control, no proto-origin guesswork, and each carries a `DEF OBS_*`. The robot block spawns at `terrain_height + 0.03` (wheels rest on the surface). The **forest** (`forest()`) has **no ring around the clearing**: (1) a **density-graded scatter** on a grid where each candidate (outside `boundary_r + FOREST_GAP`) is kept with probability `exp(−edge_dist/FOREST_TAU)` (`edge_dist` = distance to the nearest square side) → sparse near the clearing, **denser toward the map edge**; (2) a **fine dense belt** (`EDGE_BELT_GRID` < 1.3 m grid, `EDGE_BELT_W` wide) along the entire square perimeter whose overlapping trunk collisions form an **impassable wall** — flood-fill-verified to stop the Pioneer well inside the perimeter, so the robot **cannot fall off the map** (**no rocks/mountains**; lower `EDGE_BELT_GRID` toward 1.0 for more margin, raise it toward 1.25 for fewer trees). All forest `Pine` are un-`DEF`'d so they aren't goal obstacles. Also emits `DEF Pioneer3AT Pioneer3at` with the full sensor `extensionSlot` (4-layer Lidar `lidar`, `imu/gps/gyro/accelerometer`) and `DEF GOAL_TARGET` (flat red ground disk). Runs a self-validation (balanced braces, required DEFs/devices present) and prints object/vertex counts. Re-run after editing the terrain model, the clearing shape, or the scatter.

### [plot_metrics.py](diploma_webots_project/controllers/rl_scout/plot_metrics.py) — figures
Offline matplotlib script: learning curve, success rate, collision/rollover rate, episode length, TD-loss, epsilon decay, eval trajectories over the obstacle map (per-tag `obstacles{tag}.csv` with fallback), steps-to-goal histograms, **`eval_comparison.png`** (grouped bar chart of done-reason rates across DQN-random / DQN-centre / DQN-unseen / reactive-baseline summaries — whichever exist, **with 95% binomial CI error bars**), **`eval_success_by_distance.png`** (success rate per start→goal distance bin, 0–15/15–30/30–45/45+ m — the effective-range figure), and **`eval_spl.png` + `logs/spl_summary.csv`** (`compute_spl`/`plot_spl`: **SPL — Success weighted by Path Length**, Anderson et al. 2018, per eval set; SPL = mean of Sᵢ·lᵢ/max(pᵢ,lᵢ) with lᵢ = **Euclidean** start→goal — a lower bound on the obstacle-aware optimum, so reported SPL ≤ true → conservative; pᵢ = logged path length, so SPL covers only episodes with a saved trajectory; the CSV also lists per-success path-efficiency lᵢ/pᵢ. Run-5 figures: SPL 0.83 random / 0.86 centre / 0.67 unseen / 0.30 baseline, path-efficiency ~92–96% for the DQN. **Caveat: with the archived run-5 data only the first 20 trajectories/eval were logged (`save_traj` was 20), and that subset is luckier than the full 100 — `spl_summary.csv` shows both subset and full success so it's transparent; `save_traj` is now 100 so any re-eval gives full-population SPL.**) → `figures/`. Trajectory files are matched by exact regex `eval_traj{tag}_<n>.csv` (the naive glob used to mix centre/baseline trajectories into the random figure); the trajectory figure draws only the first 20 trajectories for legibility (SPL is **not** capped). Forces UTF-8 stdout (cp1252 consoles crashed on `ă`).

### [world.wbt](diploma_webots_project/worlds/world.wbt) — GENERATED, do not hand-edit
Produced by `generate_world.py` (`basicTimeStep 32`). A 150×150 m `ElevationGrid` terrain with a **Grass** appearance (rolling green hills everywhere + flat building pads, **no rim/basin**), `DEF GOAL_TARGET`, **~1560 objects**: ≈270 interior — **2 buildings** on flat pads (`Warehouse`/`Barn`) with abandoned vehicles + street furniture, **5 inaccessible obstacle zones** (boulder fields, a pine thicket, a container pile, a rubble heap), and `Oak`/`Cypress`/`SimpleTree`, `Rock`, `OilBarrel`, plus custom-`Solid` crates/logs/containers/rubble scattered over the interior; and **~1290 `Pine`** forming the forest — sparse near the clearing, **density increasing toward the map edge**, ending in a fine **impassable belt at the square perimeter** (so the robot can't fall off) — plus `DEF Pioneer3AT`. (If the Webots GUI is slow, raise `FOREST_GRID`/`EDGE_BELT_GRID` to thin the forest.) Friction is symmetric (single `coulombFriction`, no `forceDependentSlip`) because `ElevationGrid` bounding objects reject asymmetric friction. Object EXTERNPROTOs use the official `https://raw.githubusercontent.com/cyberbotics/webots/R2025a/projects/...` scheme — identical to Webots' own bundled worlds — because the local Webots install ships **no** `.proto` files (`webots://` does not resolve → "corrupt"); the robot uses the local `../protos/Pioneer3at.proto`. Webots needs internet on first load to fetch/cache the object PROTOs. Only PROTOs that actually exist in the R2025a tree are referenced (`Pine` under `objects/trees`, `OilBarrel` under `objects/obstacles`, `StreetLight`/`TrafficCone` under `objects/traffic`, `FireHydrant` under `objects/street_furniture`, `Rock` under `objects/rocks`, vehicles under `vehicles/protos/<brand>/`); crates, logs and industrial containers are plain `Solid` (`Box`/`Cylinder`) nodes with explicit z. Interior obstacle nodes carry `DEF OBS_*` (rocks/barrels/containers/crates/logs → matched by `DEF_PREFIX_RADIUS`); **all forest `Pine` are intentionally left un-`DEF`'d and absent from `PROTO_RADIUS` so they don't count as goal obstacles** (goals stay in the clearing regardless). `DEF Pioneer3AT` and `DEF GOAL_TARGET` are required by the controller. Device names: `front/back left/right wheel`, `lidar`, `imu`, `gps`, `gyro`, `accelerometer`.
